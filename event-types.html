Codex Session Event Types

基于本地 EVENT_TYPES.md 与 upstream Codex rollout policy 整理。核心规则：先看外层 record.type，再看内层 payload.type。

三层模型

查看器中的 eventType 等于 payload.type || record.type。这方便筛选，但会隐藏同名 payload 在不同 record 下的语义差异。

1. 持久化结构层 session_meta、turn_context、world_state、compacted 等。用于恢复、fork、重放状态，不是对话消息。
2. 模型可见 item 层 record.type=response_item。这些是模型历史和恢复上下文的核心 source of truth，包括消息、reasoning、工具调用和工具返回。
3. legacy/UI 事件层 record.type=event_msg。用于 UI 回放、状态提示和兼容事件流。不是所有实时事件都会写入 rollout。

典型 turn 链路

实际文件可能省略实时 begin/delta 事件
turn_context本轮环境快照：cwd、sandbox、model、timezone。
event_msg / user_message用户在 UI 中提交的消息，常用作 turn 边界。
response_item / message(user)真正进入模型上下文的 user 材料包，可能包含注入上下文。
event_msg / task_startedturn 开始边界，和 task_complete 配对。
response_item / reasoning模型 reasoning item，内容经常加密或只保留 summary。
response_item / *_call模型下发工具调用，通过 call_id 配对结果。
response_item / *_output工具返回给模型的结果，MCP 输出也会转换成 function_call_output。
event_msg / task_completeturn 完成边界，可能带 duration 和最后助手消息。
不要只看 payload.type 判断语义。最典型的坑是 agent_message：在 response_item 下是多 agent 通信的模型可见 item；在 event_msg 下是助手消息的 UI/legacy 文本投影。
